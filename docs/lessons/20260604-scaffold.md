# Scaffold Lessons

Things learned while bootstrapping the project. None of these are
unique to a single task — they are project-wide.

## Yorkie SDK

### Use `@yorkie-js/sdk`, not `yorkie-js-sdk`

The current Yorkie JS SDK ships under the scoped package
`@yorkie-js/sdk` (with `@yorkie-js/react` for hooks). The older
unscoped `yorkie-js-sdk` on npm is outdated and points to v0.6, which
does not match the API examples in
[`yorkie-js-sdk/examples`](../../../yorkie-js-sdk/examples/).

**Why:** When the scaffold first ran `pnpm add yorkie-js-sdk`, it
resolved to the legacy package. Importing `YorkieProvider` from
`@yorkie-js/react` then pulled in the *new* `@yorkie-js/sdk` as a
transitive dep — leaving two SDK versions in `node_modules` and
breaking type checks on `useDocument`.

**How to apply:** When adding Yorkie deps, always use the `@yorkie-js/*`
scoped names. If you see `yorkie-js-sdk` (unscoped) anywhere in
`package.json`, replace it.

### `VITE_YORKIE_API_KEY` is required at runtime

Without it, `<Room>` short-circuits to a setup hint screen instead of
calling `YorkieProvider`. The build does *not* fail — the app just
refuses to attach to any document.

**Why:** Failing loudly at the room boundary keeps the lobby usable
during setup, and makes the missing-env case unambiguous in the UI.

**How to apply:** When debugging "the canvas screen never appears
after creating a room," check `.env` before suspecting the network or
the SDK.

## Canvas rendering

### Full redraw on every doc change is intentional

[`src/Canvas.tsx`](../../src/Canvas.tsx) wipes the canvas and replays
every stroke on each `useEffect` triggered by the strokes array.

**Why:** Strokes are bounded per round (cleared at round start), and
the game design caps stroke count at `playerOrder.length *
turnsPerPlayer` per round. Incremental rendering would require local
diff bookkeeping that adds complexity for no measurable win at this
scale.

**How to apply:** Don't optimize this until profile data says
otherwise. If a future feature breaks the bound (freeform whiteboard
mode, persistent doodling between rounds), revisit then.

### Touch input goes through Pointer Events + `touch-action: none`

The canvas uses `onPointerDown / Move / Up` (not separate
`onTouchStart` etc.) and the CSS class sets `touch-action: none` to
suppress the browser's default scroll/pan gestures.

**Why:** Pointer Events unify mouse, touch, and stylus into one event
stream. Adding parallel touch handlers would double-fire on devices
that emulate mouse events from touch, producing duplicate strokes.

**How to apply:** When adding new input affordances on the canvas
(eraser, color picker tied to drag), keep using Pointer Events.
Resist adding `onTouch*` handlers.
