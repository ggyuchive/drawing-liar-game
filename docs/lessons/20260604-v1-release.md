## Real-time / Yorkie

### Turn timers must be measured locally, never off a shared wall-clock

The auto-advance first computed remaining time as
`docStoredTurnStartedAt + turnTimeMs - Date.now()`. `turnStartedAt`
was written with the *writer's* `Date.now()`; every other client
subtracted it from *their own* clock. Across two devices with even a
few seconds of clock skew the result went negative, the turn fired
instantly, and â€” because each advance re-anchored from another
skewed clock â€” every turn cascaded straight into voting.

**Why:** A document-stored absolute timestamp is only comparable on
the clock that wrote it. CRDT sync gives you the same *value*
everywhere, not the same *clock*.

**How to apply:** Measure durations from the moment **this client
observes** the turn (key a `setTimeout(turnTimeMs)` on a change of
`turnStartedAt`/`turnIndex`), and use the document field only as a
"new turn" *signal*, not as arithmetic input. `TurnTimer` does the
same for its display. Never `Date.now() - remoteTimestamp` for
gameplay timing.

### Identity must be an app-owned stable id, not the Yorkie actor/client id

All game state (playerOrder, scores, colors, votes, hostId) was first
keyed on `doc.getChangeID().getActorID()` / presence `clientID`.
Those are **connection-scoped**: a reload or re-attach gets a new one,
so every stored reference orphaned â€” the reloaded player dropped out
of their own game and rendered as `???`.

**Why:** The actor id identifies a *connection*, not a *user*. Yorkie
doesn't carry a stable per-user identity across attaches for you.

**How to apply:** Generate a per-tab `uid` (sessionStorage â€” survives
reload and reconnect within a tab, unique per tab), put it in
presence, and key **all** game state on it. See `getSessionUid` and
`dedupeByUid` in `src/util.ts`. This is also written up as an SDK
issue draft (`yorkie-js-sdk/ISSUE-presence-identity-on-reconnect.md`).

### A reconnect leaves a ghost presence for a moment

Right after reconnect the presence list briefly holds both the old
(disconnecting) and new connection of the same user â†’ the player
shows up twice. `dedupeByUid` (last-wins) collapses them.

### Per-tab vs per-browser storage

The display name lived in `localStorage`, which is **shared across
tabs of one browser** â€” so testing several players as tabs made them
clobber each other's names, and a reload picked up the wrong one. The
active name now lives in `sessionStorage` (per tab); `localStorage` is
only a prefill default for a fresh tab. Same lifetime reasoning as the
`uid`.

## Build / tooling

### The Yorkie SDK is a single ~700 kB chunk â€” sub-300 kB isn't reachable

`manualChunks` splits app / React / Yorkie, but the SDK is one
dependency that can't be carved smaller without dynamic-importing the
whole real-time layer. We set `chunkSizeWarningLimit` above it (with a
comment) rather than pretend the budget target was met.

### React Compiler lint rules shape time-based components

`react-hooks/purity`, `/refs`, and `/set-state-in-effect` reject
`Date.now()` during render, ref read/write during render, and
`setState` directly in an effect body. A local countdown therefore
reads time **only inside the interval effect** and exposes derived
state â€” not a ref read in render. Reach for that shape for any
clock-driven component.

### Keep correctness logic in pure modules for CI

Host election, the 8-player cap, color assignment, turn rotation,
spectator/duplicate-name predicates, and scoring live in pure
functions (`src/game/engine.ts`, `state.ts`) that `Room.tsx` drives
from inside `update()`. That let `multiuser.test.ts` simulate whole
multi-player games (Vitest, no server, no browser) and is the
regression guard for the "jumps to voting early" class of bug. Don't
bury such logic inside React effects or `update()` bodies.

### A pasted Code of Conduct can trip a content filter

Generating the full Contributor Covenant text (its enumerated
prohibited-behavior list) was blocked by an output content filter.
Referencing the Covenant by URL with a short pledge + contact works
and is just as valid.
